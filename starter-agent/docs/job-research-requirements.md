# Starter Agent 求职调研外部能力需求

## 需求背景

Starter Agent 已具备以下相关能力：

- `search_jobs_serpapi`：通过 SerpAPI 搜索公开岗位摘要和来源链接。
- `search_job_description`：使用自建 HTTP 抓取器读取 JD，但实际效果不理想。
- 知识库与 RAG：支持 Markdown 文档入库、检索、引用和基于证据回答。
- 文件式简历工具：包括 `read_resume` 等，但不等同于知识库 RAG 检索。

本次以新方案为准，引入 Playwright MCP 读取公开 JD 页面，并通过 `job-research` Skill 编排 SerpAPI、Browser MCP、RAG 与知识库入库流程。

旧 `search_job_description` 及其抓取、解析代码后续可以在确认已被新能力完整替代后删除或停用。本阶段不删除任何旧能力。

## 功能范围

### 范围内

- 使用独立 JSON 文件配置 MCP Servers。
- 兼容并真实加载以下 Playwright MCP 配置：

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest"
      ]
    }
  }
}
```

- 建立 MCP Server Registry、能力发现、健康检查、启停和单 Server 刷新能力。
- 全量发现 Playwright MCP 提供的 Tools、Resources 和 Prompts。
- 对 MCP Tool 进行逐个启用、评审和 allowlist 管理。
- 建立统一的后端 Pre-Tool-Call Gate。
- 在聊天中展示 Tool 调用确认卡并阻塞等待用户决定。
- 使用 SerpAPI 查找岗位摘要和公开来源链接。
- 使用 Playwright MCP 访问公开 JD 页面并提取完整内容。
- 使用现有知识库 RAG 从用户简历中检索可引用证据。
- 完整 JD 经用户确认后写入知识库。
- 新增 `job-research` Skill，规定调用顺序、验证要求和停止条件。
- 提供统一的「能力管理」入口，包括 `MCP Servers` 和 `Skills` 两个页签。
- 保留正常运行和 MCP 不可用时的真实运行记录。

### 范围外

- 自动登录招聘网站。
- 自动投递岗位。
- 自动填写或提交申请表。
- 自动上传简历。
- 自动发送站内消息或邮件。
- 绕过验证码、登录墙、付费墙、robots 限制或其他访问控制。
- 将 API Key、Token、Cookie、邮箱授权码或个人登录信息写入仓库。
- 仅使用 Mock、静态页面或模型口述结果完成验收。
- 第一阶段删除旧 JD 抓取代码。
- 第一阶段编写技术设计或实施计划。

## 目标用户与使用场景

### 目标用户

需要使用 Starter Agent 搜索岗位、阅读公开 JD、根据个人简历证据分析匹配度，并将确认后的 JD 保存到知识库的求职者。

### 使用场景

1. 用户输入岗位关键词、地点和期望数量。
2. Agent 使用 SerpAPI 返回岗位摘要和公开来源链接。
3. 用户选择岗位，或 Agent 根据 Skill 规则选择候选链接。
4. Agent 使用 Playwright MCP 访问公开岗位详情页。
5. Agent 提取岗位名称、公司、地点、职责、必备要求和来源 URL。
6. Agent 验证页面是否为具体岗位详情页、内容是否完整、来源是否可追溯。
7. Agent 使用 RAG 从用户已入库简历中检索与 JD 相关的原文证据。
8. Agent 展示 JD、匹配证据、缺口和全部 Tool Call 结果。
9. 用户确认完整 JD 后，系统将其作为 `job_description` 类型写入知识库。
10. 外部能力不可用时，Agent 明确说明缺失能力和未完成步骤，不补写或猜测内容。

## 用户故事

- 作为求职者，我希望搜索公开岗位并看到来源链接，以便核验岗位真实性。
- 作为求职者，我希望 Agent 能读取岗位详情页，而不只依赖搜索摘要。
- 作为求职者，我希望知道 Agent 调用了哪些外部工具、输入了什么以及得到什么结果。
- 作为求职者，我希望 Agent 只引用简历中真实存在的内容，不编造经历。
- 作为求职者，我希望在完整 JD 写入知识库前先查看并确认。
- 作为管理员，我希望查看每个 MCP Server 的配置、运行状态、版本和能力。
- 作为管理员，我希望只刷新一个 MCP Server，而不影响其他 Server。
- 作为管理员，我希望逐个启用、关闭和评审 MCP Tool。
- 作为管理员，我希望关闭的 Tool 不再向模型暴露完整定义。
- 作为管理员，我希望高风险 Tool 调用必须经过后端权限检查和人工确认。
- 作为管理员，我希望所有确认、取消、超时和实际执行都有审计记录。
- 作为维护者，我希望新 Playwright 方案验证通过后能够清理旧 JD 抓取实现。

## 功能需求

### FR-1 MCP 配置加载

- MCP Server 使用独立 JSON 配置文件。
- 配置文件支持原生 `mcpServers` 结构。
- 独立 JSON 配置文件的最终路径应在设计阶段根据项目现有配置加载约定确定，不得臆造为现有路径。
- Starter Agent 后端必须真实读取配置并启动所配置的 MCP Server。
- 配置中的环境变量或敏感凭证只能引用安全的外部存储，不得明文进入仓库。
- 配置解析失败时必须返回具体 Server、字段和错误原因。

### FR-2 Playwright MCP 接入

- 必须支持通过 `npx @playwright/mcp@latest` 启动 Playwright MCP。
- 必须完成 MCP 初始化握手并记录协议、配置版本、运行时实际解析版本和 transport。
- 必须全量发现该 Server 提供的 Tools、Resources 和 Prompts。
- 全量发现不等于全量暴露给模型，也不等于全量自动放行。
- 系统必须保存最近一次成功的能力快照及其获取时间。

### FR-3 Server Registry

每个 MCP Server 至少记录：

- Server 名称和稳定标识。
- 配置来源。
- 配置版本。
- 运行时实际解析版本。
- command、args 和 transport 的非敏感摘要。
- 启用状态。
- 连接状态。
- 健康状态。
- 最近检查时间。
- 已发现 Tools、Resources 和 Prompts 数量。
- 当前能力快照版本。
- 快照是否过期。
- 最近错误。
- Tool allowlist。
- 正在运行的任务数量或占用状态。

### FR-4 单 Server 刷新

- 支持按单个 Server 重新检查和刷新。
- 刷新时重新发现该 Server 的 Tools、Resources、Prompts、Schema、版本和健康状态。
- 不得要求同时刷新全部 Server。
- 不得断开、刷新或修改其他 Server。
- 不得影响其他 Server 上的运行中任务。
- 刷新成功后生成新的不可变能力快照。
- 刷新失败时保留上一份可用快照，将其标记为过期并显示真实错误。
- 刷新过程中已开始的调用使用其开始时绑定的快照；下一轮模型请求必须使用刷新后的最新有效快照。

### FR-5 Server 与 Tool 启停

- Server 和 Tool 都必须有独立的启用状态。
- Server 关闭后，其全部能力不得作为 callable tools 暴露。
- Tool 关闭后，不得进入模型 callable tools。
- 关闭项只允许在轻量能力目录中保留名称、所属 Server 和启用状态。
- 关闭项的完整 Description、Input Schema、Output Schema 或其他全量定义不得注入模型 Context。
- Tool 重新启用后，必须经过连接状态、健康状态、评审和权限策略检查，才能重新暴露。
- 启停操作必须由真实后端持久化，页面刷新后状态保持一致。

### FR-6 Tool 评审与暴露

- Playwright MCP 的全部 Tool 必须先发现并查看真实 Schema。
- 默认不得将全部 Tool 自动暴露给模型。
- 只有已连接、已启用且通过暴露评审的 Tool，才允许把完整 Name、Description 和 Input Schema 加入模型 callable tools。
- 未评审或未通过暴露评审的 Tool 不得由模型自主调用。
- 用户从管理页面对未评审 Tool 发起测试，或明确请求调用某个未评审 Tool 时，后端可为这次精确调用建立确认请求；确认不改变该 Tool 的长期评审状态。
- 明确评审为禁止的 Tool 不得通过普通聊天确认变为 callable tool。
- 启停、评审或刷新后，下一轮模型请求必须使用最新快照。

### FR-7 Allowlist

- Tool 自动执行 allowlist 以单个 Tool 为最小粒度。
- 初始状态下没有任何 Playwright Tool 自动加入 allowlist。
- 必须查看真实 Tool Schema 后才能逐个加入 allowlist。
- 通过暴露评审但未加入 allowlist 的 Tool，每次调用都必须暂停并请求用户确认。
- 加入 allowlist 的普通 Tool 可自动执行，但仍必须通过 Pre-Tool-Call Gate。
- 初始网络协议范围允许 `http://` 和 `https://`。
- 后续可以从设置页面配置更细的域名或资源范围。
- `file://` 等非 HTTP(S) 页面不得作为公开 JD 来源。
- Tool allowlist 不得绕过动作级强制确认或禁止规则。

### FR-8 Pre-Tool-Call Gate

所有内置 Tool 和 MCP Tool 的真实外发调用前，都必须经过统一的后端 Gate。Gate 至少检查：

- Server 是否存在、连接、健康且启用。
- Tool 是否存在且启用。
- Tool 所属能力快照是否有效。
- Tool 是否通过相应评审。
- 调用参数是否符合当前快照中的真实 Schema。
- URL、协议、域名和资源范围是否符合当前策略。
- Tool 是否位于自动执行 allowlist。
- 是否命中禁止动作或强制人工确认规则。
- 待外发数据是否包含简历原文、Cookie、Token、个人信息或其他敏感内容。
- 用户确认是否存在、有效且与当前调用参数完全绑定。
- 运行预算、超时和重复调用限制是否满足。

前端确认按钮只能提交用户决定，不能自行判定或绕过 Gate。

### FR-9 人工确认

非 allowlist Tool 的确认卡至少显示：

- Server 和 Tool 名称。
- 参数摘要。
- 目标 URL、域名或资源。
- 风险说明。
- 待外发数据摘要及去向。
- Tool 是否经过评审。
- 是否属于强制确认动作。
- 「仅本次执行」。
- 「执行并加入白名单」。
- 「取消」。

确认行为必须满足：

- 用户确认前不得发送真实 MCP Tool 请求。
- “仅本次执行”仅授权参数完全相同的单次调用。
- “执行并加入白名单”仅适用于允许持久放行的 Tool。
- 强制确认动作不得通过加入 allowlist 绕过。
- 确认请求支持超时、用户取消和服务端取消。
- 确认处理必须防止重复提交和重复执行。
- 确认状态必须具有审计记录。
- 页面刷新或连接恢复后，仍能显示有效的待确认状态。
- 取消或超时后不得执行 Tool。
- 确认后若 Server、Tool、Schema、参数或快照发生变化，旧确认失效。

### FR-10 Browser 动作边界

始终禁止：

- 登录或创建招聘网站账号。
- 填写或提交求职申请。
- 上传简历或其他文件。
- 发送站内消息、邮件或其他外部信息。
- 绕过验证码、登录墙、付费墙、robots 限制或访问控制。
- 使用获取到的 Cookie、Token 或个人凭证访问页面。

始终强制人工确认：

- 文件下载。
- 执行任意页面脚本。
- 写入浏览器存储。
- 修改远端状态。
- 其他可能产生外部副作用但未被明确禁止的动作。

允许逐个 Tool 评审后加入 allowlist 的只读行为包括：

- 打开 HTTP(S) 页面。
- 读取 DOM 或无障碍快照。
- 提取页面文本。
- 滚动页面。
- 点击“展开完整 JD”“下一页”“Cookie 同意”等不提交用户数据、不改变远端业务状态的控件。

实际权限必须依据 Playwright MCP 返回的真实 Tool Schema 确定，不能只按推测的 Tool 名称分类。

### FR-11 SerpAPI 岗位搜索

- 使用现有工具 `search_jobs_serpapi`。
- 实际输入 Schema 为：

```json
{
  "query": "string，必填，2–300 字符",
  "location": "string，可选，最长 100 字符",
  "limit": "integer，可选，1–10，默认 5"
}
```

- 返回结果必须包含可核验的岗位摘要、来源、来源 URL 和检索时间。
- 搜索结果仅作为线索，不得直接视为完整 JD。
- SerpAPI 不可用时必须停止搜索并展示真实错误。

### FR-12 完整 JD 读取与验证

- Browser MCP 仅访问公开 HTTP(S) 求职页面。
- 必须区分岗位搜索列表页和单个岗位详情页。
- 完整 JD 至少提取：

  - 岗位名称。
  - 公司。
  - 地点。
  - 职责。
  - 必备要求。
  - 来源 URL。
  - 读取时间。
  - 完整性状态。

- 必须保存或引用真实 Tool Result，不能只保存模型整理后的文本。
- 页面内容过长时应分段读取并保留来源关系。
- 无法确认内容完整时必须标记为“不完整”，不得进入知识库。
- 页面拒绝访问时不得绕过限制。
- SerpAPI 摘要与实际页面冲突时，以已读取的公开来源页为准，并显示差异。

### FR-13 RAG 简历证据检索

- `job-research` 必须使用现有知识库检索能力从用户简历中获取证据。
- 不得将旧文件工具 `read_resume` 直接视为 RAG。
- 应为 Skill 提供名称和 Schema 明确的知识库检索能力，或提供等价、可审计的调用入口。
- 检索应限定为用户当前作用域内的简历文档。
- 输出证据至少包含连续原文引用、文档、版本、Chunk 标识及定位信息。
- 没有足够证据时必须返回缺口或拒绝形成对应结论。
- RAG 不可用时仍可返回已验证 JD，但必须跳过简历匹配并说明原因。

### FR-14 JD 确认入库

- 只有状态为“完整且已验证”的 JD 才能进入确认流程。
- 确认卡必须展示结构化 JD 预览和来源 URL。
- 用户必须逐条明确确认是否入库。
- 用户确认后，以 `job_description` 文档类型写入当前用户的知识库。
- 入库内容必须保留来源 URL、读取时间和必要的来源元数据。
- 用户取消、确认超时或连接中断时不得入库。
- Tool 读取成功不等于用户同意入库。
- 入库成功后必须返回文档、版本和索引状态。
- 入库或索引失败时必须显示真实错误，不得声称已保存。
- 重复 URL 或相同内容再次入库时必须提示重复或走明确的版本更新流程，不得静默创建不可区分的副本。

### FR-15 `job-research` Skill

Skill 必须规定：

1. 何时调用岗位搜索。
2. 何时要求用户补充关键词或地点。
3. 如何从 SerpAPI 结果中选择公开来源 URL。
4. 调用 Browser MCP 前如何检查 Server、Tool 和权限。
5. 如何读取并验证完整 JD。
6. 何时停止继续浏览。
7. 如何调用 RAG 检索简历证据。
8. 如何将每条匹配结论关联到真实简历引用。
9. 如何展示 Tool 输入、输出、来源和失败。
10. 如何请求用户确认 JD 入库。
11. 外部服务不可用、页面拒绝访问、内容过长或证据不足时如何降级。
12. 哪些情况下不得继续或不得声称任务完成。

Skill 不得隐藏、吞并或伪造 Tool Call。

### FR-16 能力管理界面

前端提供统一的「能力管理」入口，并包含：

- `MCP Servers` 页签。
- `Skills` 页签。

所有状态来自真实后端配置、Registry 和运行状态。后端不可用或操作失败时必须显示错误，不得在前端伪造成功状态。

### FR-17 MCP Servers 页签

页面至少支持：

- 查看 Server 列表。
- 查看配置来源。
- 查看配置版本和运行时实际解析版本。
- 查看 transport。
- 查看启用、连接和健康状态。
- 查看已发现 Tools、Resources 和 Prompts。
- 查看能力快照版本及是否过期。
- 查看逐 Tool 启用、评审和 allowlist 状态。
- 查看最近错误。
- 在权限控制下连接、断开、启用、禁用和重新检查。
- 单独刷新一个 Server。
- 查看 Tool 的真实 Description 和 Schema，但仅向有管理权限的用户展示。
- 修改 Tool allowlist。
- 修改 HTTP(S) 域名或资源范围设置。
- 操作后重新读取后端状态并保持一致。

### FR-18 Skills 页签

页面至少展示：

- Skill 名称。
- 描述。
- 来源。
- 版本或更新时间。
- 文件位置。
- 启用状态。
- 工具依赖。
- 触发示例。
- 不触发示例。
- 详情。

页面支持：

- 启用。
- 禁用。
- 重新加载。
- 查看详情。

Skill 禁用或依赖不可用时，下一轮模型请求必须反映最新状态。

### FR-19 审计与运行记录

至少记录：

- MCP 配置加载。
- Server 启停和连接变化。
- 能力发现与单 Server 刷新。
- 快照版本变化。
- Tool 启停、评审和 allowlist 变化。
- Gate 判定。
- 确认创建、确认、取消、超时和失效。
- Tool 实际调用开始与结束。
- Tool Result 来源。
- JD 入库确认与结果。
- 错误类型、时间和关联调用标识。

审计日志不得记录 API Key、Token、Cookie、密码、授权码或不必要的简历全文。

## 非功能需求

### 安全性

- 所有 Tool 调用必须经过统一后端 Gate。
- 前端不得拥有最终授权决定权。
- 敏感凭证不得进入模型 Context、Tool Result、日志或仓库。
- 关闭的 Server 或 Tool 不得泄漏完整能力定义。
- 外部网页内容视为不可信输入，不得将页面指令当作系统指令执行。
- Tool Result 在进入模型 Context 前必须执行长度和敏感数据控制。

### 一致性

- 配置、Registry、能力快照、模型 callable tools 和管理页面状态必须一致。
- 启停、评审、allowlist 或刷新完成后，下一轮模型请求使用最新快照。
- 页面刷新后不得恢复为仅存在于前端内存中的旧状态。

### 可用性

- 单个 MCP Server 故障不得导致其他 Server 不可用。
- 刷新失败时保留上一份可用快照。
- 外部能力不可用时提供明确降级信息。
- 确认等待状态支持刷新恢复、取消和超时。

### 可观测性

- 能区分配置失败、进程启动失败、初始化失败、发现失败、浏览器启动失败、Tool 调用失败、页面拒绝访问、解析失败、内容不完整和入库失败。
- 错误信息包含可操作原因，但不泄漏敏感数据。
- 能通过 Context 调试信息检查实际发送给模型的 callable tools。

### 性能与资源

- Server 刷新和健康检查具有超时。
- Tool 调用具有独立超时。
- 页面过长时采用受控分段处理。
- 防止相同 Tool 和相同参数被模型无限重复调用。
- 一个 Server 的慢操作不得阻塞其他 Server 的管理操作。

### 兼容性

- 兼容 Windows PowerShell 环境。
- 兼容项目当前 Python、FastAPI、Pydantic 和 YAML 配置体系。
- 独立 MCP JSON 与现有主 YAML 配置职责明确。
- 不臆造仓库中不存在的 CLI 命令、配置路径或 Tool 名称。

## 验收标准

### AC-1 能力管理入口

前端存在统一的「能力管理」入口，并以 `MCP Servers` 和 `Skills` 两个页签分别管理外部连接和程序性能力。

### AC-2 Server 状态真实可见

`MCP Servers` 页签能显示 Server 列表、来源、配置版本、运行时解析版本、transport、启用状态、连接状态、健康状态、能力列表、allowlist、快照状态和最近错误。数据来自真实后端。

### AC-3 单 Server 刷新

刷新 Playwright Server 时只重新发现 Playwright 的能力与健康状态，不影响其他 Server 或运行中任务。刷新失败时保留上一份可用快照并明确标记过期和错误。

### AC-4 Tool 启停与 Context 隔离

每个 Tool 均可启用或关闭。关闭 Tool 后：

- callable tools 中不存在该 Tool。
- 模型请求中不存在其完整 Description 和 Input Schema。
- 轻量目录中最多保留名称、所属 Server 和启用状态。

重新启用并通过策略检查后，下一轮模型请求才恢复完整 Schema。

### AC-5 真实 Context 验证

必须通过真实模型请求捕获或 Context 调试信息证明 Tool 关闭后的 Schema 已移除、重新启用后才恢复。不能只检查前端开关。

### AC-6 Pre-Tool-Call Gate

所有真实 MCP 调用均经过统一后端 Gate。allowlist 中的普通 Tool 可自动执行；非 allowlist Tool 显示参数、风险和数据去向并等待确认。

### AC-7 确认选项

确认卡提供：

- 仅本次执行。
- 执行并加入白名单。
- 取消。

强制确认动作不得通过加入白名单绕过。

### AC-8 确认阻塞

在用户确认前，MCP Server 没有收到真实 Tool 请求。取消或超时后同样没有真实请求。重复提交不会造成重复执行。

### AC-9 Skills 管理

`Skills` 页签能查看名称、描述、来源、版本或更新时间、文件位置、启用状态、工具依赖、触发示例和不触发示例，并支持启用、禁用、重新加载和查看详情。

### AC-10 后端一致性

管理操作调用真实后端接口。操作失败时显示错误；刷新页面后状态与后端配置、Registry 和运行状态一致。

### AC-11 Playwright 真实启动

Starter Agent 能真实加载独立 MCP JSON 配置；`npx @playwright/mcp@latest` 对应进程能启动、完成 MCP 初始化并返回真实能力列表。

### AC-12 Schema 评审

在查看真实 Playwright Tool Schema 前，没有 Tool 被默认全部暴露或加入 allowlist。

### AC-13 Browser 边界

Browser MCP 只能处理允许的公开 HTTP(S) 求职页面，不会自动登录、投递、发送信息、上传简历或绕过站点限制。

### AC-14 完整 JD 提取

从至少一个真实公开 JD URL 提取岗位名称、公司、地点、职责、必备要求和来源 URL，并通过真实 Tool Result 核对。

### AC-15 端到端真实调用

必须持续排查配置、进程启动、初始化、能力发现、浏览器运行、Gate、Tool 调用和结果解析问题，直到真实公开 JD 调用跑通。配置完成、单元测试通过、Mock 成功或模型声称成功均不能作为完成依据。

### AC-16 Skill 编排

`job-research` Skill 能按顺序组合 `search_jobs_serpapi`、Playwright MCP 和知识库 RAG，并展示每次 Tool Call 的输入、输出、来源和失败。

### AC-17 RAG 引用

岗位匹配结论具有来自用户简历知识库的真实引用。证据不足时明确标记缺口，不生成虚假经历。

### AC-18 JD 确认入库

完整 JD 读取后先展示预览和来源 URL。用户确认前知识库中不存在该 JD；确认后以 `job_description` 类型成功入库并完成索引。

### AC-19 降级行为

SerpAPI、MCP、RAG 或页面读取不可用时，Agent 明确说明缺少的能力和未完成步骤，不伪造 JD 或简历证据。

### AC-20 运行记录

保留：

- 一条完整成功记录：搜索、浏览、JD 验证、RAG 检索、用户确认和知识库入库。
- 一条 MCP 不可用记录：包含真实错误、降级输出和未执行步骤。

### AC-21 旧能力替换评估

只有在 Playwright 端到端真实验收完成、功能覆盖明确且回归测试通过后，才能提出停用或删除旧 `search_job_description` 及重复抓取组件。删除必须单独确认。

## 边界情况

- SerpAPI 返回聚合页而不是具体岗位页。
- 来源 URL 已过期、跳转、删除或改变岗位。
- 页面需要登录、验证码、Cookie 或地区确认。
- 页面使用客户端渲染、懒加载、iframe 或分页。
- “展开完整 JD”按钮无法点击或会触发外部副作用。
- 页面同时包含多个岗位。
- 岗位标题、公司或地点与 SerpAPI 摘要不一致。
- 页面正文极长，超出 Tool Result 或模型 Context 限制。
- 页面只提供 PDF、图片或下载文件。
- MCP 进程启动后立即退出。
- MCP 初始化成功但能力发现失败。
- 刷新期间 Server 配置发生变化。
- 刷新返回的新 Schema 与旧 Schema 不兼容。
- 用户确认期间 Tool 被禁用、刷新或移除。
- 确认请求重复提交或浏览器页面刷新。
- 同一个 Tool 已加入 allowlist，但本次参数触发强制确认规则。
- Tool Result 中包含页面注入指令、Cookie、Token 或个人信息。
- RAG 中没有简历、简历版本不正确或没有相关证据。
- JD 已存在于知识库，或同一 URL 内容发生变化。
- 用户确认 JD 后，入库成功但索引失败。
- SerpAPI 可用但 Playwright 不可用。
- Playwright 可用但 RAG 不可用。
- 一个 MCP Server 卡死，而其他 Server 仍有运行任务。

## 风险与待确认事项

1. 独立 MCP JSON 的具体文件路径尚未确定，应在设计阶段根据项目现有配置加载方式决定。
2. 初始允许全部 HTTP(S) 会扩大 SSRF、重定向和非公开资源访问风险；设计阶段需明确私网地址、localhost、云元数据地址和重定向目标的处理规则。
3. Tool 级 allowlist 可能不足以区分同一 Tool 的只读参数和高风险参数，因此 Gate 仍需执行参数级动作检查。
4. Playwright MCP 使用 `@latest` 会造成运行时版本变化；必须记录实际解析版本并处理 Schema 漂移。
5. 未评审 Tool 不向模型暴露完整 Schema，但管理页测试或用户明确调用仍需要安全的精确参数构造与确认机制。
6. 当前 RAG 是后端 API/聊天模式，不是模型 callable tool；后续需要确定供 Skill 使用的明确接口和 Schema。
7. 当前知识库仅接受允许扩展名的 Markdown 上传；JD 自动入库需要复用授权、版本、索引和容量限制。
8. 旧 `search_job_description` 与 Playwright MCP 存在能力重叠，删除范围必须在真实替代验证后确定。
9. Playwright 的实际 Tool 名称和 Schema 必须以运行时发现结果为准，本需求不预设不存在的 Tool 名称。
10. 公开招聘网站可能限制自动访问；系统必须遵守站点限制，不将访问失败视为可绕过问题。
