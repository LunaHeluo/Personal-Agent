# 求职邮件工具套装需求文档

## 需求背景

求职者在主动投递、请求内推、联系招聘方、面试后致谢、跟进进度、沟通 Offer、拒绝或撤回申请等场景中，需要频繁撰写和管理邮件。不同场景对事实依据、语气、时机、地区礼仪和附件要求不同，用户容易遇到以下问题：

- 不知道如何针对具体职位、公司和联系人撰写专业邮件。
- 邮件内容模板化，未能准确体现职位要求与用户真实经历的匹配点。
- 多岗位并行申请时，邮件线程、联系人、申请状态和跟进时间难以管理。
- 中英文表达、不同国家或地区的招聘礼仪及语气把握困难。
- 发送前可能遗漏主题、称呼、附件、联系方式，或残留占位符和错误的公司、岗位信息。
- 为提高说服力而产生未经用户材料支持的技能、经历、学历、成绩或关系描述。
- 草稿、保存和发送之间的状态不清晰，可能造成误发或让用户误以为邮件已经发送。

本能力面向 Starter Agent 的求职场景，定位为“求职邮件工具套装”，覆盖邮件材料获取、内容生成与改写、发送前检查、草稿保存、人工确认后发送、申请记录和跟进提醒等完整流程。

工具套装需要提供统一入口，根据用户意图识别邮件场景、判断所需信息是否充分，并协调相应子能力。该统一入口只负责编排和状态说明，不得绕过用户确认执行外部发送。

## 功能范围

### 范围内

- 支持以下求职邮件场景：
  - 主动投递职位。
  - 请求熟人、校友或从业者内推。
  - 主动联系招聘方或用人经理。
  - 面试后的感谢邮件。
  - 申请或面试进度跟进。
  - Offer 相关沟通，包括确认收到、提问、协商和接受。
  - 拒绝 Offer、拒绝继续流程或撤回申请。
- 生成新邮件以及基于已有邮件线程生成回复。
- 对用户已有邮件进行改写、润色、压缩、扩写和风险检查。
- 从用户输入、已有用户档案、简历、作品集、职位网页和邮件线程中获取必要上下文。
- 读取用户提供的职位网页地址并提取职位描述及来源信息。
- 支持中文、英文及中英双语输出。
- 根据目标国家或地区适配招聘邮件礼仪。
- 允许用户指定语气、长度、语言和正式程度。
- 信息不足时优先向用户提出补充问题；用户无法或不愿补充时，使用醒目的占位符。
- 生成主题、收件人建议、邮件正文、附件清单、发送时机建议和发送前检查结果。
- 保存邮件草稿。
- 在用户查看最终内容并明确确认后发送邮件。
- 记录申请、联系人、邮件往来状态和下次跟进时间。
- 创建、更新和取消跟进提醒。
- 检查附件是否存在及是否符合发送要求。
- 检测敏感信息、不合适措辞、夸大或无依据陈述、上下文矛盾及残留占位符。

### 范围外

- 未经用户明确确认自动发送单封或批量邮件。
- 在没有邮箱授权的情况下声称已经访问、保存到邮箱或发送邮件。
- 自动注册邮箱、代替用户完成登录或规避邮箱服务的安全验证。
- 将邮箱密码、授权码、OAuth 凭据或 API key 写入代码、文档、日志或版本控制。
- 虚构用户的技能、经历、学历、成绩、工作成果、雇佣状态或与联系人之间的关系。
- 保证招聘方回复、面试邀请、Offer 或求职成功。
- 依据受保护特征生成歧视性、欺骗性或骚扰性沟通内容。

## 目标用户与使用场景

### 目标用户

- 首次求职、缺少专业邮件经验的应届生。
- 同时申请多个职位、需要管理沟通状态的社招候选人。
- 转行或跨地区求职、需要解释可迁移能力的候选人。
- 使用非母语撰写中文或英文求职邮件的用户。
- 需要针对不同国家或地区调整沟通礼仪的跨境求职者。

### 典型使用场景

1. 用户粘贴职位网页地址，要求根据已有简历生成英文主动投递邮件。
2. 用户请求一封中文内推邮件，并指定简洁、礼貌但不过度正式的语气。
3. 用户提供面试邮件线程和面试中的讨论要点，要求生成感谢邮件。
4. 用户询问何时跟进尚未回复的申请，并保存一封待发送草稿及提醒。
5. 用户收到 Offer 后要求生成澄清问题或薪资协商邮件。
6. 用户希望撤回申请，要求生成简短、保持职业关系的邮件。
7. 用户要求直接发送邮件；系统展示最终收件人、主题、正文和附件，取得明确确认后才发送。
8. 用户没有提供个人信息时，系统尝试从已有用户档案获取；档案仍不充分时向用户提问。

## 用户故事

- 作为求职者，我希望根据具体职位和我的真实经历快速生成邮件，以减少从空白开始撰写的时间。
- 作为求职者，我希望系统读取职位网页并提取关键要求，以便邮件与当前职位保持一致。
- 作为同时申请多个岗位的用户，我希望记录申请、联系人和邮件状态，以便知道下一步该做什么。
- 作为需要跟进的用户，我希望获得合理的发送时机建议并创建提醒，以避免遗漏或过度催促。
- 作为中文或英文非母语用户，我希望指定语言、语气和正式程度，以获得符合目标地区习惯的表达。
- 作为谨慎的用户，我希望系统只使用有来源的个人经历，不替我夸大或虚构事实。
- 作为发送邮件的用户，我希望在发送前看到完整内容、收件人和附件，并由我明确确认，以防误发。
- 作为未授权邮箱的用户，我希望仍能生成和保存本地草稿，而不会被要求在代码中提供密码或密钥。
- 作为重视隐私的用户，我希望知道哪些个人资料和邮件内容会被保存，并能够查看、更正和删除这些数据。
- 作为收到招聘方来信的用户，我希望系统理解当前邮件线程并生成上下文一致的回复，而不是创建脱离上下文的新邮件。

## 功能需求

### FR-01 场景识别与能力编排

- 系统必须识别用户是在新建、回复、润色、检查、保存、发送、记录还是安排跟进。
- 系统必须识别主动投递、内推请求、招聘方联系、面试感谢、进度跟进、Offer 沟通、拒绝和撤回场景。
- 场景不明确且会影响邮件内容时，系统必须先向用户确认，不得自行选择高风险意图。
- 统一入口必须清楚展示本次执行的结果状态，例如“仅生成内容”“草稿已保存”“等待发送确认”或“已发送”。

### FR-02 上下文与资料获取

- 系统必须支持用户直接输入或粘贴个人信息、职位描述、联系人信息和邮件线程。
- 用户未提供必要个人资料时，系统应从已获用户授权的现有用户档案中查找。
- 同时存在用户本次输入与已有档案时，应以用户本次明确输入为准；发现冲突时必须向用户指出并请求确认。
- 系统必须支持读取用户提供的职位网页地址，提取公司、职位名称、职责、要求、地点及原始网页地址。
- 网页无法访问、需要登录、内容动态加载失败或职位已下线时，系统必须说明失败原因并请求用户粘贴职位内容，不得臆测职位要求。
- 系统必须记录职位信息的来源和获取时间，使用户能够人工复核。
- 读取邮件、网页、用户档案或外部文件前，必须具备相应授权；缺少授权时应返回明确的授权需求，不得假装已读取。

### FR-03 信息完整性处理

- 系统必须根据邮件场景检查必需信息，包括发送目的、公司或组织、职位或机会、联系人、可依据的个人经历、期望语言及已有线程。
- 缺少会显著影响内容的信息时，系统必须优先提出简短、具体的补充问题。
- 用户表示无法提供、跳过或不再回答时，系统可以继续生成，但必须使用醒目的占位符，例如 `[招聘方姓名待补充]`。
- 含未替换占位符的邮件不得进入发送确认，必须先由用户补齐或明确确认如何处理。

### FR-04 岗位匹配与事实依据

- 系统必须从职位描述和用户材料中提取与当前邮件目标相关的匹配点。
- 邮件中关于用户的事实必须能够追溯到用户本次输入、已有用户档案、简历、作品集或用户确认过的内容。
- 系统可以重组、概括和润色已有事实，但不得新增无依据的技能、经历、学历、成绩、成果数字、任职状态或人际关系。
- 缺少某项职位要求的证据时，系统应将其视为信息缺口，不得将推断写成事实。
- 发现用户材料之间存在矛盾时，必须标记矛盾并请求用户确认。

### FR-05 邮件生成与改写

- 系统必须为所有范围内场景生成新邮件或线程回复。
- 输出必须包含：
  - 建议主题。
  - 收件人及抄送建议；未知信息应明确标记。
  - 邮件正文。
  - 建议附件清单。
  - 建议发送时机。
  - 发送前检查结果。
- 系统必须支持中文、英文和中英双语版本。
- 系统必须支持用户指定简洁或详细程度、正式程度，以及正式、自然、热情、克制等语气。
- 未指定语言或语气且无法从上下文可靠判断时，系统必须先询问；用户不回答时可采用与当前对话或邮件线程一致的语言和职业化中性语气，并明确说明。
- 生成线程回复时，必须结合最近邮件内容、发件人角色、已回答问题、待处理事项和线程语气，避免重复询问或答非所问。

### FR-06 地区礼仪适配

- 系统必须允许用户指定目标国家或地区。
- 系统应根据已知地区调整称呼、直接程度、日期表达、礼貌表达、跟进间隔建议和落款习惯。
- 地区未知且会显著影响内容时，系统必须询问用户；用户不回答时采用通用职业表达并说明未进行特定地区适配。
- 地区适配不得覆盖用户明确指定的语气和事实边界。

### FR-07 发送前检查

- 系统必须检查以下项目：
  - 收件人、公司名、职位名和联系人称呼是否一致。
  - 主题是否存在且与场景一致。
  - 正文是否包含未替换占位符。
  - 提到的附件是否实际存在，实际附件是否符合当前邮件目的。
  - 联系方式是否缺失或格式明显异常。
  - 是否存在无材料依据、夸大、误导或互相矛盾的个人陈述。
  - 是否泄露不必要的敏感信息。
  - 是否包含贬低现任或前任雇主、过度催促、冒犯、骚扰或明显不专业的表达。
  - 回复邮件是否遗漏对方提出的关键问题或行动项。
- 检查结果必须按“通过”“警告”“阻止发送”分类，并给出具体问题位置及修改建议。
- 收件人无效、缺少正文、存在占位符、缺少用户确认、附件与正文声明不符或存在未经确认的事实时，必须阻止发送。

### FR-08 草稿保存

- 系统必须支持保存邮件草稿，并明确区分“本地草稿”和“邮箱服务中的草稿”。
- 没有邮箱授权时，可以保存本地草稿，但不得声称已保存到用户邮箱。
- 草稿结果必须包含草稿标识、保存位置或可检索引用、保存时间、当前状态和“尚未发送”的明确说明。
- 用户修改草稿后，发送前必须针对最新版本重新执行发送前检查。

### FR-09 邮件发送与人工确认

- 每次发送前，系统必须向用户展示最终收件人、抄送人、主题、正文、附件和检查结果。
- 只有用户针对该最终版本作出明确发送确认后，系统才可以调用外部发送能力。
- 用户此前的笼统授权、偏好设置或“以后都不用问”不得替代对当前邮件最终版本的确认。
- 草稿内容、收件人或附件在确认后发生任何变化时，原确认失效，必须重新确认。
- 发送结果必须来自邮箱服务或发送工具的具体结果，不得根据模型文本判断发送成功。
- 发送失败时必须返回可判断的失败状态和原因，不得报告为已发送。
- 系统不得自动批量发送；多封邮件必须让用户逐封查看并确认。

### FR-10 申请记录与邮件状态

- 系统必须能够创建和更新申请记录，至少包含公司、职位、职位来源、申请时间、联系人、邮件场景、草稿或发送状态、最近沟通时间、当前申请状态和下次跟进时间。
- 系统必须将“内容已生成”“草稿已保存”“等待确认”“发送中”“已发送”“发送失败”等状态明确区分。
- 系统必须支持用户查询、修正和删除申请记录。
- 同一公司或职位可能存在多次申请时，系统必须避免静默覆盖，并提示用户确认关联到现有记录还是创建新记录。

### FR-11 跟进建议与提醒

- 系统必须能根据邮件场景、上次沟通时间、招聘方给出的时间承诺、目标地区和用户偏好提出跟进时机建议。
- 招聘方已给出明确回复日期时，建议不得早于该日期，除非用户确认存在新的紧急情况。
- 系统必须支持创建、修改、取消和查看跟进提醒。
- 没有日历或提醒服务授权时，系统可以记录建议时间或提供本地提醒，但不得声称已写入外部日历。
- 提醒触发后不得自动发送邮件，只能提示用户处理或生成待确认草稿。

### FR-12 附件处理

- 系统必须支持用户指定简历、求职信、作品集或其他附件。
- 发送前必须验证被正文提及的附件是否存在、是否可读取、文件名是否明显错误，以及附件类型是否符合用户当前意图。
- 系统应提示附件版本可能与当前职位不匹配、文件名不专业或包含明显敏感信息的情况。
- 系统不得在未经用户选择的情况下自行添加用户档案中的文件作为发送附件。

### FR-13 授权、凭据与隐私控制

- 系统不得默认用户已提供邮箱、网页、档案、文件、日历或其他外部服务授权。
- 缺少授权时，系统必须明确说明受影响能力，并允许用户继续使用不依赖该授权的生成、检查或本地保存功能。
- 密码、邮箱授权码、OAuth 凭据和 API key 不得写入代码、文档、日志、草稿、申请记录或版本控制。
- 系统必须让用户能够查看、更正和删除被持久化的个人资料、邮件内容、联系人及申请记录。
- 系统必须对日志和错误信息中的邮件正文、附件内容、联系方式和凭据进行最小化处理，不得默认记录完整敏感内容。

## 非功能需求

### 安全与权限

- 外部读取、外部写入和邮件发送必须有清晰的权限边界及可辨认的执行状态。
- 邮件发送属于高影响外部操作，必须执行逐封、逐版本人工确认。
- 授权失败、授权过期或权限不足时应安全失败，不得降级为绕过授权的操作。
- 凭据必须由安全配置或授权机制提供，不得硬编码或进入版本控制。

### 隐私

- 只获取和保存完成当前功能所必需的数据。
- 持久化数据必须能关联其来源，并支持用户查看、更正和删除。
- 不应在普通运行日志中记录完整邮件正文、简历正文、附件内容、邮箱地址或电话；诊断所需信息应脱敏。
- 数据保留周期、静态数据保护方式、备份删除策略和跨服务数据传输规则必须在上线前明确。

### 可靠性与可追溯性

- 每次读取职位网页应保留来源地址和获取时间。
- 每封草稿和最终发送版本必须能够区分，避免将旧确认用于新版本。
- 外部操作必须返回明确、可机读的成功、失败、等待授权或等待确认状态。
- 用户必须能够从结果中判断邮件是否真正发送，而不依赖自然语言猜测。

### 易用性

- 补充问题应优先询问对成稿影响最大的缺失信息，避免一次提出大量低价值问题。
- 检查结果必须指出具体问题和可执行的修正建议。
- 用户即使不连接邮箱、日历或用户档案，也应能通过手动输入完成邮件生成和检查。
- 中英文输出应符合职业沟通习惯，不应机械逐句翻译。

### 性能

- 对仅使用已提供文本的单封邮件生成或检查，在正常运行条件下应在 10 秒内返回结果，不含模型或外部服务不可控延迟。
- 职位网页、邮箱、档案或日历等外部读取超时后，应在 30 秒内返回明确状态，并允许用户改用手动输入继续。

### 可测试性

- 每项外部操作必须能够通过模拟结果或受控测试账号验证成功、失败、未授权和超时路径。
- 事实依据、占位符、发送确认、附件一致性和状态区分必须能通过确定性检查或人工 Review 验证。
- 不得以“模型表示已完成”“模型认为内容合适”作为验收证据。

## 验收标准

### AC-01 核心场景覆盖

使用包含必要事实的固定测试材料，分别请求主动投递、内推、招聘方联系、面试感谢、进度跟进、Offer 沟通、拒绝和撤回邮件。每个场景均应生成主题、收件人建议、正文、附件清单、发送时机和检查结果，且人工 Review 确认邮件目的与场景一致。

### AC-02 中英文及用户偏好

对同一测试场景分别指定中文、英文和中英双语，并指定不同语气、长度和正式程度。输出语言及风格应符合指令；双语版本中的公司、职位、时间、事实和行动请求必须一致，可通过字段对照和人工 Review 判断。

### AC-03 职位网页读取

给定可公开访问的固定职位网页，工具结果应包含公司、职位、职责、要求、地点、原始地址和获取时间。抽取结果应与网页可见内容逐项比对。给定失效、需登录或无法读取的网页时，系统应返回失败原因并要求用户粘贴内容，不得生成声称来自网页的职位事实。

### AC-04 用户资料获取与冲突

当用户未输入个人资料但存在已授权用户档案时，生成内容只使用档案中的事实。当本次输入与档案冲突时，系统必须指出具体冲突并等待确认，不得静默选择。没有输入、档案或授权时，系统必须提问，不得虚构。

### AC-05 信息不足与占位符

删除测试材料中的联系人和职位名称后请求生成邮件，系统应先提出补充问题。用户明确跳过后，邮件必须包含醒目占位符；发送前检查必须将其判定为阻止发送。

### AC-06 事实依据

为测试职位加入用户材料中不存在的技能、成果数字和人际关系要求。生成邮件不得将这些内容写成用户事实。人工逐句核对邮件中的个人陈述，均应能定位到输入材料或用户确认记录；无法定位的内容应被检查功能标记并阻止发送。

### AC-07 邮件线程回复

给定包含两个问题和一个截止时间的固定邮件线程，生成回复必须回答或明确处理两个问题，并正确保留截止时间。检查结果应指出任何未回答的行动项。

### AC-08 地区礼仪

对同一跟进场景分别指定两个目标国家或地区，结果应在称呼、日期格式、直接程度或礼貌表达上体现适配，并保持事实一致。由熟悉对应地区职业沟通的人工 Reviewer 按预先定义的礼仪检查表验收。

### AC-09 发送前检查

分别构造公司名不一致、错误职位名、缺少主题、残留占位符、附件缺失、未经证实的成果和不专业措辞案例。系统必须定位具体问题并分类；占位符、无正文、无效收件人、附件声明不符和未经确认事实必须阻止发送。

### AC-10 草稿状态

在没有邮箱授权时保存草稿，结果必须明确说明其为本地草稿、尚未发送，并提供可检索的草稿引用。在具有测试邮箱授权时保存邮箱草稿，必须由邮箱服务结果验证草稿存在，且测试收件箱中没有已发送邮件。

### AC-11 逐封发送确认

用户首次要求“直接发送，以后不用问”时，系统仍必须展示最终版本并进入等待确认状态。未确认时调用记录中不得出现发送操作。用户确认后发送测试邮件，成功状态必须与邮箱服务返回结果及测试邮箱中的已发送记录一致。

### AC-12 版本变化后重新确认

用户确认一封测试邮件后，在发送前改变收件人、正文或附件。系统必须使原确认失效并再次请求确认；第二次确认前不得调用发送能力。

### AC-13 发送失败

模拟授权过期、无效收件人、服务超时和服务端拒绝。系统必须返回对应失败或未知状态，不得显示“已发送”，并保留可重试或修改的草稿。

### AC-14 申请记录

创建两条同公司、不同职位的申请并更新其中一条的沟通状态。查询结果必须正确区分两条记录。尝试重复创建同公司同职位记录时，系统必须提示关联或新建，不得静默覆盖。

### AC-15 跟进提醒

当招聘方承诺在固定日期回复时，系统不得建议在该日期前跟进。创建、修改和取消测试提醒后，通过提醒工具或记录查询确认状态变化。没有日历授权时不得声称外部日历事件已创建。

### AC-16 附件检查

正文声明附有简历但未提供文件时，检查必须阻止发送。提供可访问附件后，结果应列出文件名并通过存在性检查。附件无法读取、版本明显不符或文件名含不专业占位文本时，系统必须给出警告或阻止发送。

### AC-17 隐私与凭据

使用唯一测试标记作为模拟邮箱地址、邮件正文片段和凭据，执行生成、读取失败及发送失败流程。扫描应用日志、草稿元数据和申请记录：凭据必须零命中，完整敏感正文不得出现在普通日志；用户执行删除后，相关可查询持久化记录应不存在。

### AC-18 效率与错误减少

使用覆盖核心场景的固定验收样本，由测试人员记录从提交完整材料到获得可发送草稿的时间和人工修改项：

- 至少 90% 的样本一次生成后包含全部规定输出项。
- 至少 95% 的预置严重错误被发送前检查识别并阻止发送。
- 相比同一测试人员手工从空白起草的基线，中位成稿时间至少降低 30%。
- 所有样本均不得出现无依据的个人事实或未经确认的发送。

效率指标仅用于验收流程表现，不用于承诺招聘结果。

## 边界情况

- 职位网页已下线、被地区限制、需要登录、禁止抓取或包含多个不同职位。
- 网页职位名称与招聘邮件中的职位名称不一致。
- 用户同时提供多个职位、多个简历版本或多个联系人，但未说明当前目标。
- 用户档案缺失、过期、与本次输入冲突，或用户撤销了档案访问授权。
- 用户只提供招聘方姓名，没有邮箱地址；或邮箱格式合法但实际不可投递。
- 邮件线程内容不完整、转发层级混乱、混合多种语言或包含多个职位。
- 招聘方要求在特定日期前不要跟进，或已明确拒绝进一步联系。
- 用户要求夸大技能、编造推荐关系、伪造在职状态或隐藏重要事实。
- 用户要求群发、密送大量招聘方，或对同一联系人进行高频跟进。
- 用户在确认后修改正文、附件、收件人或抄送人。
- 草稿保存成功但外部邮箱发送失败，或发送请求超时导致最终状态暂时未知。
- 正文提到附件但文件不存在、文件损坏、格式不支持、版本错误或包含敏感信息。
- 同名公司、同名职位、重复申请或同一联系人服务多个职位。
- 时区、节假日、夏令时或地区差异导致发送时机和提醒时间存在歧义。
- 用户要求中英双语，但两种语言中的数字、日期、职位或行动请求不一致。
- Offer 沟通涉及法律、税务、签证或薪酬判断；系统只能辅助起草，不替代专业意见。
- 用户删除申请记录后，外部邮箱或日历中仍可能保留由其服务独立管理的数据。
- 外部服务授权过期、权限不足、速率限制、暂时不可用或返回部分成功。

## 风险与待确认事项

### 已识别风险

- **误发风险：** 发送属于不可轻易撤销的外部操作，必须逐封、逐版本确认，并清楚区分草稿与已发送状态。
- **事实失真风险：** 模型可能把职位要求误写成用户能力，所有个人陈述都必须有可追溯依据。
- **隐私风险：** 简历、邮件、联系人、申请记录和附件可能包含敏感个人信息，需限制采集、日志和保存范围。
- **凭据泄露风险：** 邮箱和外部服务接入可能涉及授权凭据，任何凭据都不得进入代码、文档、日志或版本控制。
- **网页内容风险：** 职位网页可能失效、过期、内容不完整或与实际招聘要求不同，必须保留来源和获取时间供人工复核。
- **地区礼仪误判风险：** 同一地区内部也存在行业和公司差异，适配结果只能作为沟通辅助，必要时需要人工 Review。
- **提醒可靠性风险：** 本地提醒、外部日历和应用状态可能不同步，必须显示提醒实际保存位置和状态。
- **成功指标偏差：** 回复率和求职结果受候选人背景、市场和岗位等外部因素影响，不作为本能力的直接验收标准。

### 待确认事项

1. 首版需要接入哪些邮箱服务，例如 Gmail、QQ 邮箱、Outlook 或通用 IMAP/SMTP？
2. “草稿保存”是否要求同时支持本地草稿和邮箱服务草稿，还是允许首版先实现其中一种？
3. 申请记录和邮件正文分别保留多长时间？是否允许用户设置保留周期？
4. 用户执行删除时，是否要求同时删除本地备份、搜索索引和外部服务中的对应内容？
5. 持久化数据是否有加密、设备范围、跨设备同步或云端存储要求？
6. “已有用户档案”当前包含哪些字段、存储在哪里，以及读取时采用何种用户授权规则？
7. 职位网页读取允许访问哪些网站？遇到网站登录、付费墙、反自动化限制或服务条款限制时应采用何种产品策略？
8. 首版需要支持哪些国家或地区，地区礼仪检查表由谁维护和 Review？
9. 跟进提醒需要接入哪些渠道，例如 Starter Agent 内部提醒、系统通知或外部日历？
10. 附件检查需要支持哪些文件格式、最大文件大小，以及是否需要检查文件正文中的敏感信息和职位匹配度？
11. Offer 薪资协商是否只负责邮件表达，还是还需要外部市场数据支持？若需要外部数据，数据来源和时效要求尚未明确。
12. 性能验收中的 10 秒和 30 秒目标是否符合当前模型与外部服务运行环境，需要在基准环境确定后复核。
13. 效率验收的人工起草基线、测试人员、样本数量及严重错误样本集需要在测试前固定，避免验收口径漂移。
