# Starter Agent 带引用的个人知识库 / RAG 需求文档

## 文档定位与开发流程

本次 RAG 功能固定采用以下流程：

```text
rag-requirements.md
→ rag-design.md
→ rag-task.md
→ 按 Task 顺序执行
→ 最终验证
```

本文件定义需求与验收边界。需求确认后进入设计；不得从需求文档直接跳到代码实施。

## 需求背景

Starter Agent 当前主要通过对话上下文、长期记忆和工具调用完成求职任务。用户的简历版本、岗位 JD、面试准备笔记、公司调研资料和邮件沟通摘要通常分散在多个文件中，存在以下问题：

- 查找和组合资料耗时，跨文档比较困难。
- 多份简历或 JD 并存时容易混淆版本。
- 模型可能将常识、推测或岗位要求误写成用户的真实经历。
- 回答缺少可定位来源，用户无法核对结论。
- 文档更新或删除后，旧内容可能继续影响回答。
- 仅凭“回答看起来合理”无法证明检索和引用正确。

本功能为 Starter Agent 增加一个由用户管理的私人知识库，使 Agent 能够检索用户授权的求职文档，严格基于检索证据回答，并返回可定位到原文的引用。

知识库与长期记忆相互独立：

- 长期记忆保存少量、稳定、经确认的用户事实。
- 知识库保存可检索的文档、版本、Chunk 和来源信息。
- 知识库内容不得未经用户确认自动转化为长期记忆。
- 删除知识库文档后，长期记忆不得继续保留由该文档自动推断的事实。

## 功能范围

### 范围内

第一阶段包括：

- Starter Agent 前端提供明确、可直接访问的“知识库”入口。
- 提供独立的知识库页面或视图。
- 支持用户选择并上传 Markdown 文档。
- 显示文档上传、解析和索引状态。
- 查看已入库文档列表及基本元数据。
- 查看每份文档的 Chunk 数量和 Chunk 预览。
- 更新指定文档并使旧 Chunk 失效。
- 硬删除指定文档及其全部可检索内容。
- 根据用户问题检索当前授权范围内的知识库。
- 基于检索结果生成回答。
- 为回答中的事实性结论提供可定位引用。
- 在无充分证据时明确拒答。
- 支持独立测试 Retrieval 和 Generation。
- 支持使用无敏感信息的固定文档进行自动化和人工验收。

适合进入知识库的资料：

- 用户拥有并授权使用的 Markdown 简历版本。
- 用户保存的公开岗位 JD 或自行整理的 JD。
- 用户本人编写的面试准备笔记。
- 来源明确的公开公司调研摘要。
- 经用户确认、已去除不必要第三方隐私的邮件沟通摘要。
- 求职进度、项目经历和技能证据的用户自有记录。
- 为测试专门编写、不包含真实敏感信息的样本文档。

### 范围外

第一阶段不包括：

- PDF、DOCX、图片、扫描件或压缩包解析。
- 自动扫描用户设备或目录并批量入库。
- 未经确认自动导入邮箱、网页、云盘或其他外部系统内容。
- 自动把聊天、工具结果或长期记忆写入知识库。
- 知识库内容的公开发布或未授权跨用户、跨项目检索。
- 根据模型推测自动修正文档。
- 自动发送邮件、投递简历或执行其他外部操作。
- 将 RAG 回答视为法律、税务、签证、医疗或财务专业意见。
- 保证获得面试、Offer 或其他求职结果。
- 仅依靠生成回答的流畅度判断功能是否通过验收。

禁止上传或保存：

- 身份证、护照、银行卡等证件原件或完整号码。
- 未经授权的公司内部资料、客户资料、面试题库或保密材料。
- 真实 API Key、访问令牌或 OAuth 凭据。
- 私人邮箱密码、邮箱授权码或其他账户密码。
- 与求职任务无关且没有必要保存的健康、财务、家庭等高度敏感信息。
- 未经相关人员授权且未做必要脱敏的私人邮件全文、联系方式或第三方个人资料。

## 目标用户与使用场景

### 目标用户

- 同时申请多个岗位、需要管理多份资料的求职者。
- 拥有多份简历版本、容易混淆版本差异的用户。
- 需要基于自身真实经历准备面试或求职材料的用户。
- 希望核验 Agent 回答来源、避免虚构经历的用户。
- 希望私人资料由自己控制、可以查看、更新和删除的用户。

### 典型使用场景

1. 用户上传一份脱敏简历和一份目标岗位 JD，询问最匹配的经历并核对引用。
2. 用户上传面试准备笔记，询问其中记录的系统设计重点。
3. 用户比较两份简历版本中某项经历的表述差异。
4. 用户询问某项技能是否出现在自己的资料中；没有证据时 Agent 明确拒答。
5. 用户根据公司调研资料生成面试提问，并查看每个问题依据的来源。
6. 用户更新简历，确认旧版本的 Chunk 不再参与默认检索。
7. 用户删除已过期岗位 JD，确认该文档不能再被检索或用于回答。
8. 开发者单独调用 Retrieval，检查返回的 Chunk 是否正确，而不调用生成模型。
9. 开发者向 Generation 提供固定证据，检查回答是否只使用这些证据。

## 用户故事

- 作为求职者，我希望从 Starter Agent 前端直接进入知识库，以便管理求职资料。
- 作为求职者，我希望上传 Markdown 并看到明确处理状态，以便知道资料是否可以检索。
- 作为求职者，我希望查看文档列表、版本、Chunk 数量和预览，以便确认实际入库内容。
- 作为求职者，我希望根据简历和 JD 提问并获得引用，以便核验匹配结论。
- 作为求职者，我希望 Agent 在资料没有答案时明确拒答，以免把推测误当成事实。
- 作为拥有多份简历的用户，我希望引用能够区分文件和版本，以免混用经历。
- 作为重视隐私的用户，我希望明确知道资料保存在哪里、由谁处理以及是否会发送到外部服务。
- 作为重视隐私的用户，我希望删除文档后，其正文、Chunk、索引和关联元数据被同步清除。
- 作为工具使用者，我希望公开岗位搜索结果只有在我确认后才进入私人知识库。
- 作为开发者，我希望分别测试文档入库、Retrieval 和 Generation，以便准确定位错误。
- 作为验收人员，我希望使用固定的无敏感信息样本核对检索结果和引用，而不是依赖主观观感。

## 功能需求

### FR-01 知识库入口

- 前端必须提供文字清晰的“知识库”入口。
- 入口必须在 Starter Agent 主界面直接可见或通过稳定的一级导航直接访问。
- 入口不得仅依赖隐藏手势、开发者控制台或手工输入未展示的 URL。
- 知识库页面必须明确区别于聊天会话和长期记忆管理。

### FR-02 Markdown 文档上传

- 用户必须能够从知识库页面选择 Markdown 文件并提交上传。
- 第一阶段只接受扩展名和实际内容均符合要求的 Markdown。
- 上传前必须展示禁止上传的敏感资料类型。
- 上传前必须要求用户确认资料属于本人或已获授权。
- 系统必须限制单文件大小、文档数量和可解析文本长度。
- 系统不得允许路径穿越、可执行内容或伪装成 Markdown 的文件。
- 系统检测到明显凭据、密码、授权码或证件号码模式时必须阻止入库。
- 重复上传相同内容时，不得静默产生无法区分的重复索引。

### FR-03 入库状态

- 每次上传必须返回可识别的文档记录和入库状态。
- 状态至少应区分：待处理、处理中、已索引、失败、删除中。
- 处理阶段必须能够区分上传、解析、Normalize、Chunk 和索引。
- 上传失败、解析失败、Chunk 失败和索引失败必须提供不暴露正文的明确原因。
- 只有处于“已索引”状态的活动版本才能参与检索。
- 页面刷新后仍应能够查询文档和任务状态。

### FR-04 文档与 Chunk 查看

- 文档列表至少显示：
  - 文档标识。
  - 文件名。
  - 活动版本或内容指纹。
  - 上传时间。
  - 当前状态。
  - Chunk 数量。
- 用户必须能够查看某份文档的 Chunk 预览。
- Chunk 预览至少显示：
  - Chunk 标识。
  - 所属文档标识和文件名。
  - 文档版本或内容指纹。
  - Markdown 标题或章节路径。
  - 原文行号范围。
  - 可供人工核对的文本片段。
- 预览不得把整份私人文档写入普通应用日志。

### FR-05 Chunk 和来源元数据

- 文档必须按可复现规则切分，不能由生成模型任意决定 Chunk 边界。
- 每个 Chunk 必须保留足以回到原文的来源元数据。
- Metadata 至少包含：
  - `document_id`
  - `filename`
  - `page`
  - `section`
  - `user_id`
  - `project_id`
  - `version`
  - `created_at`
  - `chunk_id`
  - `start_line`
  - `end_line`
- Markdown 的 `page` 可以为空，但章节路径与行号范围必须存在。
- 文档正文中的指令只能被视为资料内容，不得覆盖系统规则、用户授权或工具权限。

### FR-06 Retrieval

- 系统必须根据用户问题从当前有效、已索引且在授权范围内的文档中检索相关 Chunk。
- 检索必须排除处理中、失败、已删除、无权限或不符合过滤条件的文档。
- 检索必须支持 top-k。
- 检索必须支持按文档标识、文件名、版本或资料类型限制范围。
- 权限过滤必须在检索阶段生效，不得先检索其他用户内容后再隐藏。
- Retrieval 必须能够独立调用和测试，不依赖 Generation 才能观察结果。
- Retrieval 结果至少包含：
  - 查询。
  - 命中的 Chunk。
  - 每个 Chunk 的来源元数据。
  - 可比较的相关性或排序信息。
  - `source_ref`。
  - 无命中状态。
- 检索无命中与检索系统失败必须是可区分状态。
- 不得仅用模型生成的解释代替真实检索结果。
- 第一阶段必须采用不依赖模型的确定性全文索引。
- 第一阶段不引入 Embedding、向量索引、向量数据库、混合检索或模型 Rerank。

### FR-07 Generation

- Generation 必须只把本次允许使用的检索证据作为私人文档事实来源。
- Generation 必须能够使用固定 Retrieval 结果独立测试。
- 回答中的具体经历、技能、时间、数字、岗位要求和沟通事实必须由引用支持。
- 多个事实性结论应在对应结论附近提供引用，不能只在回答末尾附一个关系不明的来源列表。
- 引用必须真实支持相邻结论，不得只因关键词相似就视为有效。
- 模型的一般性建议必须明确区别于知识库事实。
- 如果证据存在冲突，系统必须指出冲突文档和版本，不得静默选择其一。
- 如果只有部分结论有证据，系统只能回答有证据的部分，并说明其他部分证据不足。
- 模型不能自行生成 canonical 文件名、版本、行号和引用片段；最终引用必须由服务端根据实际 Chunk 组装。

### FR-08 引用

- 每个事实性结论必须引用本次 Retrieval 实际返回的 Chunk。
- 可定位引用必须包含：
  - `document_id`
  - `filename`
  - `document_version`
  - `chunk_id`
  - Markdown 标题或章节路径
  - `start_line`
  - `end_line`
  - 支持结论的原文片段
- 引用片段必须是对应原文的连续子串。
- 引用不得指向无权限、旧版本、已删除或未参与本次 Retrieval 的 Chunk。
- 文档更新后，旧 Chunk 引用必须失效，不得错误映射到新内容。

### FR-09 拒答规则

以下情况必须明确拒答、缩小回答范围或请求补充资料：

- 检索不到支持问题的证据。
- 命中内容与问题相关性不足。
- 证据不足以支持所要求的确定性结论。
- 多份文档相互冲突且无法确定有效版本。
- 用户要求虚构、夸大或隐瞒求职经历。
- 用户要求泄露凭据、密码、授权码或不必要的第三方隐私。
- 用户要求访问未授权文档、其他用户资料或已删除资料。
- 用户要求将受保护特征用于歧视性求职判断。
- 用户要求把知识库内容包装成确定的法律、税务、签证、医疗或财务结论。

拒答必须说明“知识库中没有足够证据”或具体权限、冲突原因，不得编造答案填补空缺。

如果问题可以由现有公开搜索工具回答，系统可以在明确区分“私人知识库证据”和“外部公开信息”的前提下建议调用工具；未经用户要求，不得用外部结果冒充知识库证据。

### FR-10 文档更新

- 系统必须能够区分文件名相同但内容不同的版本。
- 更新必须先完整建立新版本，再切换活动版本。
- 新版本入库失败时，旧活动版本必须继续可用。
- 新版本激活后，旧 Chunk 必须立即退出检索。
- 旧版本正文、Chunk 和索引必须按已确认策略清理。
- 引用必须始终指向回答时实际使用的版本。
- 更新冲突不得静默覆盖，必须要求用户刷新或重新确认。

### FR-11 文档删除

- 用户必须能够删除自己有权管理的文档。
- 删除采用硬删除：清除原始正文、Chunk、检索索引和关联元数据。
- 删除开始后，目标文档必须立即停止参与新的检索。
- 删除完成后，通过旧 `document_id`、`chunk_id`、文件名、关键词或相似查询均不得返回旧内容。
- 删除失败或部分失败时必须明确显示，不得提前宣称已删除。
- 删除不得误删其他文档、版本、用户或项目的数据。
- 审计记录如需保留，只能保留必要的非正文信息，不得使旧内容可恢复检索。

### FR-12 与现有能力配合

- 岗位搜索和 JD 网页抽取工具只负责公开、时效性资料；结果必须经用户确认后才能进入知识库。
- 简历工具继续读取和管理指定文件；知识库保存上传快照，两者不得静默混用版本。
- 邮件工具继续负责授权查询、草稿和发送；知识库第一阶段只接收用户主动上传且已脱敏的邮件摘要。
- 长期记忆继续保存少量稳定、经确认的事实；知识库文档不得自动写入长期记忆。
- 时间工具和公开搜索结果可补充时效性信息，但必须与私人文档引用分别标记。
- RAG 回答不得自动触发邮件发送、简历修改或其他有副作用的工具操作。

### FR-13 权限、隐私与数据处理

- 系统只能检索当前用户、项目和知识库范围内已授权且有效的文档。
- 不得跨用户、跨项目或跨未授权资料范围检索。
- 用户必须能够知道哪些文档当前会参与检索。
- 本地与外部模型处理必须由用户按知识库选择，并明确展示数据去向。
- 切换到外部处理前必须取得明确确认。
- 外部服务只能接收完成当前操作所需的最少证据，不得发送整个知识库。
- 取消授权后，相关内容必须停止参与新的处理。
- 系统不得在日志、错误、遥测或测试报告中记录完整私人文档、密码、Token 或授权码。

## 非功能需求

### NFR-01 可验证性

- Retrieval 和 Generation 必须拥有清晰输入、输出边界，可以分别测试。
- 测试应能够固定文档、Chunk、查询和预期引用。
- 验收不得仅依赖模型输出是否流畅或“看起来合理”。
- 每个关键结果必须能通过文档内容、检索记录或确定性状态检查复核。

### NFR-02 安全与隐私

- 文档内容不得出现在普通日志中。
- 所有文档和索引访问必须执行用户、项目和知识库隔离。
- 文件名、Markdown 内容和元数据均视为不可信输入。
- 文档提示注入不得改变系统规则、权限或工具调用策略。
- 系统不得把真实凭据写入源码、配置、文档或版本控制。
- 使用外部处理服务前必须明确披露数据去向。

### NFR-03 数据一致性

- 文档状态、版本、Chunk 与检索索引必须保持一致。
- 入库失败不得留下可检索半成品。
- 更新不得同时产生两个默认活动版本。
- 删除必须可检测部分失败，并支持幂等重试。
- 相同内容重复提交不得产生无法解释的检索重复。

### NFR-04 可用性

- 用户无需开发者工具即可完成上传、查看状态、预览 Chunk、更新和删除。
- 状态与错误信息使用普通用户可以理解的中文。
- 入口、上传、预览和删除具有明确标签及键盘可操作性。
- 删除前必须展示目标文档并要求确认。

### NFR-05 性能

- 上传后应及时显示已接收和处理状态。
- 文档列表和 Chunk 预览应支持合理数量文档下的交互使用。
- Retrieval、索引完成时间、单文件大小和总容量应在基准环境中设定并可配置。
- 第一阶段优先满足个人知识库规模，不要求大型分布式检索能力。

### NFR-06 可观测性

- 系统应记录不含正文的上传、解析、Chunk、索引、检索、更新和删除事件。
- 事件至少包括操作类型、文档标识、阶段、耗时、Chunk 数和错误码。
- 应能区分解析失败、索引失败、检索无命中、权限拒绝和生成失败。

### NFR-07 可维护性

- 文档管理、Retrieval、Generation 和引用校验边界必须清晰。
- 更换检索或生成实现不应改变引用、拒答和删除的外部行为。
- 测试不得依赖真实隐私资料、真实邮箱或生产凭据。

## 验收标准

### AC-01 知识库入口

- 启动 Starter Agent 前端。
- 主界面存在明确标记为“知识库”的入口。
- 用户可以直接通过该入口进入知识库页面。
- 不允许以开发者控制台或隐藏 URL 代替入口。

### AC-02 Markdown 上传与状态

- 在知识库页面选择一份无敏感信息的 Markdown 并提交。
- 页面显示上传和入库状态。
- 状态最终变为“已索引”或等价成功状态。
- 不支持格式、解析失败或索引失败必须显示明确失败状态，且文档不可检索。

### AC-03 文档列表和 Chunk 预览

- 文档成功入库后，文档列表能找到该文件。
- 列表显示文档标识、文件名、版本或指纹、状态和 Chunk 数量。
- 用户可以打开 Chunk 预览。
- 预览与原 Markdown 对应章节一致，并显示版本、Chunk、标题路径、行号和片段。

### AC-04 两份安全文档入库

- 使用两份固定、无真实敏感信息的 Markdown：
  - 一份虚构候选人的脱敏简历。
  - 一份虚构公司的公开岗位 JD 或面试准备笔记。
- 两份文档均成功上传并完成索引。
- 文档列表和 Chunk 统计分别正确，不得相互混淆。

### AC-05 有证据回答与可定位引用

- 使用固定问题询问测试文档中明确存在的事实。
- 回答内容与原文一致。
- 每个事实性结论具有真实支持该结论的引用。
- 引用包含文档、版本、Chunk、章节路径、行号范围和原文片段。
- 人工检查引用片段时，能在对应版本原文中找到。
- 引用与结论不一致、引用错误版本或仅关键词相关均判定失败。

### AC-06 无证据拒答

- 询问测试资料中不存在的 HR 邮箱、工作经历、技能或成果数字。
- 系统明确说明知识库没有足够证据。
- 不得根据常识、模型记忆或相似岗位信息编造答案。
- 如果部分问题有证据，只回答有证据部分并拒绝其余部分。

### AC-07 文档更新

- 更新一份已索引文档并记录旧 Chunk 标识。
- 新版本成功前旧版本继续可检索。
- 新版本激活后只检索新版本内容。
- 旧 Chunk 不再返回，旧引用不能错误定位到新内容。
- 新版本入库失败时旧版本不受影响。

### AC-08 文档删除后不可检索

- 记录已索引文档中的独特事实及引用。
- 删除文档并等待删除完成。
- 使用原问题、独特关键词、旧文档标识和旧 Chunk 标识再次检索。
- Retrieval 不得返回该文档内容。
- Chat 不得继续基于旧内容回答；如无其他证据必须拒答。
- 重启应用或新建会话后重复检查，旧内容仍不可检索。
- 仅从前端隐藏但仍可命中索引，判定失败。

### AC-09 Retrieval 独立测试

- 不调用 Generation，直接向 Retrieval 提交固定查询。
- 结果包含预期文档的预期 Chunk 和来源元数据。
- 无证据查询返回明确空命中。
- 已删除文档查询不返回旧 Chunk。
- 断言基于 Chunk、来源和内容，不基于模型描述。

### AC-10 Generation 独立测试

- 不执行实时检索，向 Generation 提供固定测试 Chunk。
- 回答只能使用提供的证据。
- 有证据时生成与证据一致且带引用的回答。
- 空 Chunk 或无关 Chunk 时明确拒答。
- 加入提示注入 Chunk 时，不执行其中指令或生成无依据结论。

### AC-11 权限和隔离

- 使用两个隔离测试用户或项目建立不同文档集。
- 一个范围不得查看、预览、检索、更新或删除另一个范围的文档。
- 使用未授权标识访问时不得泄露文档存在性或正文。

### AC-12 隐私和日志

- 测试文档放入唯一、非真实敏感标记。
- 完成上传、失败、检索、生成和删除流程后扫描日志与错误输出。
- 完整正文和唯一标记不得出现在不必要的日志中。
- 上传模拟 API Key、密码和授权码模式时必须阻止入库。
- 验收不得使用真实 API Key、私人邮箱密码或授权码。

### AC-13 工具与知识库来源区分

- 对同一问题同时提供知识库资料和公开搜索结果。
- 回答必须区分私人文档引用和外部公开来源。
- 外部工具结果未经用户确认不得自动成为知识库文档。
- RAG 问答不得自动触发邮件、简历写入或其他副作用。

### AC-14 验收判定原则

只有同时满足以下条件才可通过：

- 前端入口、上传、状态、列表、Chunk 预览、更新和删除均可操作。
- 两份无敏感信息文档成功索引。
- 有证据问题回答正确且引用可定位、可核对。
- 无证据问题稳定拒答。
- 更新后旧 Chunk 失效。
- 删除后旧内容在重启及新会话中仍不可检索。
- Retrieval 和 Generation分别通过独立测试。
- 日志和测试产物不存在真实秘密或不必要的私人正文。

“回答流畅”“回答看起来合理”或“模型自称已检索”均不能作为通过依据。

## 边界情况

- 空文件、只有标题或没有可索引正文的文件。
- 超过大小、字符数、文档数或 Chunk 数限制的文件。
- 扩展名为 `.md` 但实际为二进制或其他格式。
- 非 UTF-8、编码损坏或包含异常控制字符的 Markdown。
- 文件名包含路径分隔符、特殊字符或重复名称。
- 同一内容重复上传。
- 同名但内容不同的多个版本。
- 文档上传中断、解析成功但索引失败或索引部分写入。
- 更新与检索同时发生。
- 删除与索引、更新或检索同时发生。
- 删除请求重复提交。
- 文档列表已删除但索引仍残留。
- 一次查询同时命中新旧版本。
- 不同文档对同一事实给出冲突描述。
- 问题只有部分内容能够由证据支持。
- 命中 Chunk 包含关键词，但语义上不能支持结论。
- 答案跨越多个 Chunk 或多个文档。
- Markdown 表格、代码块、列表、引用块和超长章节被不当切分。
- 文档包含“忽略系统规则”“发送邮件”等提示注入内容。
- 文档正文包含伪造引用格式。
- 用户使用旧引用访问已经删除或更新的文档。
- 用户要求检索不属于自己的文档。
- 用户撤销外部处理授权后，本地仍存在对应副本。
- 外部生成服务不可用、超时或返回无效格式。
- 只有长期记忆但知识库无证据，或反之。
- 会话历史保留删除前回答；历史展示不得被当成当前有效证据。
- 用户上传包含第三方邮箱、电话或公司内部信息的邮件摘要。
- 用户询问岗位是否仍开放；旧 JD 只能说明文档内容，不能替代实时检查。
- 词法不同但含义相近的表达无法命中；系统不得因此编造答案。

## 风险与待确认事项

### 已确认决策

- 支持本地与外部模型处理，由用户按知识库选择并明确展示数据去向。
- 删除采用硬删除，清除正文、Chunk、索引和关联元数据。
- 涉及用户经历、岗位、公司资料或沟通事实时采用严格证据模式。
- 无证据或证据不足时明确拒答。
- 引用包含文档、版本、Chunk、章节路径、行号范围和原文片段。
- 文档更新后重新索引，旧 Chunk 与旧引用失效。
- 第一阶段检索采用 SQLite FTS5 `trigram`、BM25、元数据过滤和确定性字段加权。
- 第一阶段不实现 Embedding、向量索引、向量数据库、混合检索或模型 Rerank。

### 已识别风险

- **隐私泄露风险：** 简历、邮件摘要和求职记录可能包含个人及第三方敏感信息。
- **权限风险：** 用户可能上传无权保存的公司资料、面试题或私人通信。
- **外部处理风险：** 生成模型可能把证据发送给第三方服务。
- **提示注入风险：** 文档内容可能尝试改变系统规则或诱导调用工具。
- **引用失真风险：** 检索到相关文本不代表足以支持生成结论。
- **版本混淆风险：** 多份简历或 JD 并存时可能引用错误版本。
- **删除残留风险：** 数据可能残留在缓存、FTS、WAL、备份或外部服务中。
- **日志泄露风险：** 调试日志、异常栈或测试快照可能记录正文。
- **无证据生成风险：** 模型可能使用预训练知识或会话历史补全私人事实。
- **资料过期风险：** 岗位 JD、公司调研和沟通状态可能失效。
- **长期记忆污染风险：** 文档推断若写入长期记忆，删除原文后仍可能影响回答。
- **误拒答风险：** 分段、词法差异或检索阈值可能导致文档有答案但无法命中。

### 待确认事项

1. 第一阶段是否严格保持本机单用户，还是需要正式认证与多用户部署。
2. 文档正文、Chunk、索引、SQLite WAL 和备份的实际存储与清理范围。
3. 允许使用的外部生成服务、地区、保留政策和授权文案。
4. 是否要求静态加密、系统密钥管理或设备级访问控制。
5. 单文件大小、最大文档数、总容量和最大 Chunk 数的最终默认值。
6. `.md` 与 `.markdown` 是否均作为正式支持扩展名。
7. 同内容或同名文档再次上传时的默认交互。
8. 用户是否能同时保留多个可检索版本，还是只保留一个活动版本。
9. 用户可用的查询过滤项和默认检索范围。
10. 文档默认保留期及是否支持自动过期。
11. 备份、WAL 和审计日志存在时的删除保留期限。
12. 邮件摘要允许包含哪些字段，以及第三方信息的脱敏规则。
13. 岗位搜索、JD 抽取和邮件工具结果进入知识库的显式确认流程。
14. 知识库内容转为长期记忆时的确认、来源和联动删除规则。
15. 词法检索的 top-k、BM25/字段权重、同义词表和拒答阈值。
16. 已删除文档产生的旧聊天回答如何标记引用失效。
17. 是否需要文档导出、全部清空知识库和撤销外部处理授权。
18. 数据泄露、误上传或删除失败时的通知和恢复流程。
