# L7 · RAG 任务计划与执行

用途：设计确认后，先生成 `rag-task.md`，待用户确认任务计划后，再按顺序实现 RAG 能力。

---BEGIN---
你是我的 Agent 工程实现协作伙伴。请使用中文工作。

前提：
- `rag-requirements.md` 已确认。
- `rag-design.md` 已确认。
- 现在先生成 `rag-task.md`，不要立即修改代码。等我确认计划后，再执行。

目标：
为 Starter Agent 实现求职 Agent 的「带引用、可更新、可删除的个人知识库 / RAG 能力」。

`rag-task.md` 必须由有序 Task1 / Task2 / Task3 ... 组成。

每个 Task 必须包含：

- 任务目标
- 子任务
- 依赖关系
- 验收标准
- 预估复杂度

重要约束：

- 不要在 `rag-task.md` 中生成“状态”字段。
- 不要写“未开始 / 进行中 / 已完成”等静态状态文本。
- `rag-task.md` 专注任务拆解；实际执行进度由任务管理机制单独记录。
- 每个 Task 必须可以独立执行，并且必须按顺序推进。

建议任务拆分：

1. 实现文档入库基础能力：上传 API、读取文件、解析文本、保存 document metadata。
2. 实现 chunk 与 metadata：按段落或标题切分，保留页码/章节、document_id、version。
3. 实现索引与检索：支持 top-k、metadata filter、返回 source_ref。
4. 实现前端「知识库」页面：文件选择、上传、状态、文档列表、Chunk 数量、Chunk 预览和删除。
5. 实现带引用回答：把检索结果交给模型，并要求引用 chunk 来源。
6. 实现拒答策略：无可靠证据时拒答，而不是编造。
7. 实现文档删除或更新验证：删除后旧 chunk 不可检索。
8. 新增测试和验收文档：覆盖上传入口、Chunk 预览、有证据回答、无证据拒答、删除验证、retrieval/generation 分开测试。

输出 `rag-task.md` 后停止，等待我确认计划。

当我明确说“确认计划，开始执行”后，你再按 Task 顺序小步实现。每完成一个 Task，请汇报：

- 修改了哪些文件
- 运行了哪些测试
- 是否满足该 Task 的验收标准
- 剩余风险
---END---
