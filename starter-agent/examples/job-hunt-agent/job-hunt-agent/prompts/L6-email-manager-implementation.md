# L6 · 任务计划与执行

用途：设计确认后，先生成 `task.md`，待用户确认计划后，再按顺序执行。注意：`task.md` 不要写静态状态字段。

---BEGIN---
你是我的 Agent 工程实现协作伙伴。请使用中文工作。

前提：
- `requirements.md` 已确认。
- `design.md` 已确认。
- 现在先生成 `task.md`，不要立即修改代码。等我确认任务计划后，再执行。

目标：
为 Starter Agent 实现求职场景的「邮件工具套装」。

`task.md` 必须由有序 Task1 / Task2 / Task3 ... 组成。

每个 Task 必须包含：

- 任务目标
- 子任务
- 依赖关系
- 验收标准
- 预估复杂度

重要约束：

- 不要在 `task.md` 中生成“状态”字段。
- 不要写“未开始 / 进行中 / 已完成”等静态任务状态文本。
- `task.md` 专注任务拆解；实际执行进度在课堂开发过程中单独记录，不写死在任务模板里。
- 每个 Task 必须可以独立执行，并且必须按顺序推进。

建议对任务进行有必要的拆分：

1. 新增 `EmailManager`，统一管理 provider、mock fixture、草稿、发送门禁、结果裁剪和错误码。
2. 新增或改造工具：
   - `email_search`
   - `email_read`
   - `email_create_draft`
   - `email_send`
3. 注册工具到 `ToolRegistry`，并通过配置启用。
4. 更新 `.env.example` 或配置说明，但不要写真实密钥。
5. 新增设计文档和验收文档。
6. 新增测试，覆盖：
   - mock 搜索命中
   - 读取邮件详情，长正文会裁剪并带 `is_truncated / has_more`
   - 创建草稿不会发送
   - `email_send` 未批准时拒绝
   - 配置缺失时返回稳定错误码

输出 `task.md` 后停止，等待我确认计划。

当我明确说“确认计划，开始执行”后，你再按 Task 顺序小步实现。每完成一个 Task，请汇报：
- 修改了哪些文件
- 运行了哪些测试
- 是否满足该 Task 的验收标准
- 剩余风险
---END---
