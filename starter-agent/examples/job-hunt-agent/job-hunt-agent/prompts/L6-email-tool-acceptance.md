# L6 · 最终验证

用途：执行计划完成后，进行最终验证。不要让模型自称“已经完成”就算通过，必须用具体问题、工具结果、测试和人工 Review 判断。

---BEGIN---
你是我的 Agent 功能验收协作伙伴。请使用中文工作。

我们已经完成求职场景的「邮件工具套装」实现，现在需要做最终验证。

已知工具：
- `email_search`
- `email_read`
- `email_create_draft`
- `email_send`

请输出一份最终验证方案，包含四部分：

一、课堂 Chat 验收问题

至少 6 个自然语言问题，覆盖：
- 成功路径
- 失败路径
- 发送确认门禁
- 长结果裁剪
- 配置缺失
- 邮件不存在

示例方向：
- 帮我找一下最近 HR 关于 Agent 岗位的邮件。
- 读取第一封邮件，告诉我面试时间和需要准备什么。
- 帮我写一封礼貌回复，但不要发送。
- 现在尝试发送这封邮件。

二、pytest 验收命令与关键断言

列出建议运行的测试命令，并说明每个测试的关键断言。

三、人工 Review 清单

必须检查：
- Tool 是否调用 manager，而不是散落 provider 逻辑。
- provider / IMAP / SMTP / 密钥是否走配置。
- 发送是否需要人工确认。
- 长结果是否带 `is_truncated / has_more / source_ref`。
- 日志是否避免敏感信息进入。
- `requirements.md`、`design.md`、`task.md`、`.env.example` 是否同步。

四、最终通过标准

请明确列出什么情况下可以认为本功能通过，什么情况下必须退回修复。

注意：
- 不要把“模型回答说它能做到”当成验收。
- 不要要求真实发送邮件作为课堂必需项。
- 真实 Gmail / QQ 邮箱接入必须使用应用专用密码、OAuth 或 QQ 授权码，并且不能提交到仓库。
---END---
