# L7 · RAG 最终验证

用途：RAG 能力实现后，生成最终验证方案。不要把“模型回答看起来合理”当成通过，必须检查引用、拒答、删除和测试记录。

---BEGIN---
你是我的 Agent 功能验收协作伙伴。请使用中文工作。

我们已经完成求职 Agent 的「带引用个人知识库 / RAG 能力」，现在需要做最终验证。

请输出一份最终验证方案，包含四部分：

一、Chat 验收问题

至少 8 个自然语言问题，覆盖：

- 有证据回答
- 无证据拒答
- 引用能否定位原文
- 文档删除后旧内容不可检索
- 不同文档版本的检索
- 与求职案例相关的问题

示例方向：

- 根据我的简历和目标 JD，总结我最匹配的三项经历，并给出引用。
- 这份面试准备笔记里提到的系统设计重点是什么？请引用来源。
- 我的资料里有没有写某家公司具体 HR 邮箱？如果没有，请拒答。
- 删除某份 JD 文档后，再问该岗位要求，确认不能从旧索引回答。

二、pytest 或手动验收命令与关键断言

列出建议运行的测试命令或手动验收步骤，并说明关键断言。

三、人工 Review 清单

必须检查：

- 前端是否有明确的「知识库」入口。
- 上传后是否显示入库状态、文档列表、Chunk 数量和 Chunk 预览。
- chunk 是否保留 `document_id / filename / page or section / version`。
- 检索是否先做 metadata filter。
- 回答是否只基于检索证据。
- 引用是否真的支持结论。
- 无证据时是否拒答。
- 删除文档后，对应 chunk 和向量是否不可检索。
- `rag-requirements.md`、`rag-design.md`、`rag-task.md` 是否同步。

四、最终通过标准

请明确列出什么情况下可以认为本功能通过，什么情况下必须退回修复。
在验收问答的时候，可以使用真实的模型进行测试，请使用我内置好的glm-4.7进行真实链路验证。

注意：

- 不要要求上传真实隐私资料作为基础验收的必需项。
- 不要把流畅回答当成正确回答。
- RAG 验收必须同时看 Retrieval 和 Generation。
---END---
