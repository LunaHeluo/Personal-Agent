# L7 · RAG 知识库需求文档

用途：承接第 6 课流程，为求职 Agent 的个人知识库能力生成 `rag-requirements.md`。本提示词只做 brainstorming 和需求文档，不做设计、不生成任务计划、不修改代码。

---BEGIN---
你是我的 Agent 功能开发协作伙伴。请使用中文工作。

现在我们要为 Starter Agent 新增「带引用的个人知识库 / RAG 能力」。第一阶段只做 brainstorming 和 `rag-requirements.md`，不要写设计，不要生成任务计划，不要修改任何文件。

背景：
- 本次 RAG 功能使用 `rag-requirements.md → rag-design.md → rag-task.md → 执行 → 最终验证` 的流程开发。
- 本次要把 Agent 从“能调用工具做事”扩展到“能基于私人文档回答，并给出引用”。
- 案例仍然是求职 Agent：知识库可包含简历版本、岗位 JD、面试准备笔记、公司调研资料、邮件沟通摘要等。

请先做 brainstorming：

1. 这个 RAG 能力要解决什么求职问题。
2. 哪些资料适合进入知识库，哪些资料不应该上传。
3. 这个能力和现有工具套装如何配合。
4. 哪些问题应该回答，哪些问题应该拒答。
5. 需要向用户确认哪些隐私、权限和数据范围。

然后生成 `rag-requirements.md`。

`rag-requirements.md` 必须包含以下章节：

- 需求背景
- 功能范围
- 目标用户与使用场景
- 用户故事
- 功能需求
- 非功能需求
- 验收标准
- 边界情况
- 风险与待确认事项

验收标准至少覆盖：

- Starter Agent 前端提供明确、可直接访问的「知识库」入口。
- 知识库页面可以选择 Markdown，提交上传并显示入库状态。
- 文档入库后可以查看文档列表、Chunk 数量和 Chunk 预览。
- 上传并索引两份无敏感信息文档。
- 能回答有证据的问题，并返回可定位引用。
- 对无证据问题明确拒答。
- 文档删除后，旧内容不可继续检索。
- Retrieval 与 Generation 可以分开测试。

约束：

- 不要上传身份证、未授权公司资料、真实 API Key、私人邮箱密码或授权码。
- 不要把“模型回答看起来合理”当作验收。
- 如果资料范围、权限、删除策略不清楚，必须写入“风险与待确认事项”。

输出 `rag-requirements.md` 后停止，等待我确认需求。
---END---
