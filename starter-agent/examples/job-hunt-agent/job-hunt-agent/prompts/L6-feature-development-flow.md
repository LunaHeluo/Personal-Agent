# L6 · Brainstorming 与需求文档

用途：在写代码前，把“我想做一个能力”整理成 `requirements.md`。本提示词只做需求澄清和需求文档，不做设计、不生成任务计划、不修改代码。

---BEGIN---
你是我的 Agent 功能开发协作伙伴。请使用中文工作。

现在我们要为 Starter Agent 开发一个新能力。第一阶段只做 brainstorming 和需求文档，不要写设计，不要生成任务计划，不要修改任何文件。

我要开发的能力：求职邮件工具套装
 <!-- / JD 详情抓取工具 / 日历管理工具套装】 -->

请按以下流程工作：

1. 先做 brainstorming,确认之后才开始构建需求文档
   - 这个能力要帮谁解决什么问题。
   - 这个能力可能包含哪些工具或子能力。
   - 判断它更像“单一工具”还是“工具套装”。
   - 如果是工具套装，初步说明背后是否需要 manager。
   - 列出你认为必须向我确认的问题。

2. 然后生成 `requirements.md` 写入docs/requirements.md

`requirements.md` 必须包含以下章节：

- 需求背景
- 功能范围
- 目标用户与使用场景
- 用户故事
- 功能需求
- 非功能需求
- 验收标准
- 边界情况
- 风险与待确认事项

3. 输出后停止，等待我确认需求。

约束：

- 不要默认我已经提供外部账号、密钥、邮箱授权或 API key。
- 不要建议把密码、授权码或 API key 写进代码。
- 不要把“模型说可以”当成验收标准，验收必须可以通过具体问题、工具结果、测试或人工 Review 判断。
- 如果需求不清楚，必须把问题写进“风险与待确认事项”。
---END---
